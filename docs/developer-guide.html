Getting Started

Quick Start Guide

Everything you need to get the converter running on Windows in under 5 minutes.

First-time Setup

  1. Open PowerShell in the project folder Navigate to where you unzipped the project.
  2. Create the virtual environment Only needed once per machine.
  3. Activate & install dependencies Installs all required Python packages into the venv.
  4. Start the server Launches the API + serves the web UI.
  5. Open the UI in your browser The frontend and API docs are both served from the same port. 
copy
# Step 1 — navigate to project root
cd C:\path\to\redshift_to_fabric

# Step 2 — create virtual environment (once only)
python -m venv .venv

# If Activate.ps1 is blocked, run this first:
Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned

# Step 3 — activate venv
.\.venv\Scripts\Activate.ps1

# Step 4 — install dependencies (once only)
python -m pip install -r requirements.txt

# Step 5 — start the server
python run.py server
✅ Then open in browser

UI  → http://localhost:8000
API Docs → http://localhost:8000/docs

Every Subsequent Time

copy
.\.venv\Scripts\Activate.ps1
python run.py server

Architecture

How It Works

Every conversion flows through the same 7-stage pipeline regardless of whether it came from the UI, CLI, or API.

1
Split & Classify
app/parser/splitter.py
Raw SQL text is split into individual statements at semicolons (delimiter-aware, handles quotes and block comments). Each statement is classified as TABLE, VIEW, MATERIALIZED_VIEW, or SCHEMA.
2
Parse → Intermediate Representation (IR)
app/parser/table_parser.py  /  view_parser.py
Each statement is parsed into a structured IR object (TableIR or ViewIR). Redshift-specific metadata is extracted: ENCODE, DISTKEY, SORTKEY, WITH NO SCHEMA BINDING, etc.
3
Transform
app/transformer/view_transformer.py  /  table_generator.py
25+ transformation functions run in a fixed pipeline order on the IR. Schema refs are replaced with placeholders, functions are mapped, GROUP BY ordinals expanded, CAST → CONVERT, etc.
4
Validate
app/validator/validator.py
Output SQL is scanned for residual Redshift syntax (ENCODE, DISTSTYLE, BOOLEAN type, IS TRUE/FALSE, etc.). Each find reduces the confidence score and adds a warning.
5
Score & Report
app/reporter/reporter.py
Each object receives a confidence score (0–100%). Status is HIGH_CONFIDENCE ≥90%, PARTIAL ≥65%, MANUAL_REVIEW <65%, FAILED on parse error.
6
Write Outputs
app/output/generator.py
Converted SQL is written to data/outputs/<job_id>/. A Markdown report and JSON summary are written to data/reports/<job_id>/.
7
Return / Display
app/api/routes/convert.py  /  frontend/index.html
Results are returned as JSON to the API caller. The frontend renders each object as an expandable card with syntax-highlighted SQL, warnings, and download links.

Adding Your Own Schemas

Open app/core/settings.py and add entries to schema_placeholder_map. Example for the scenario with reporting, master, sales, product schemas:

copy
# app/core/settings.py — around line 57

schema_placeholder_map: dict[str, str] = {

    # ── Existing BI schemas ────────────────────────
    "bi_alefdw":     "${rs_bi_alefdw}",
    "bi_alefdw_dev": "${rs_bi_alefdw}",
    "alefdw":        "${rs_alefdw}",
    "alefdw_dev":    "${rs_alefdw}",

    # ── YOUR NEW SCHEMAS ───────────────────────────
    # Format:  "source_schema_name": "${placeholder}"
    # Keys must be lowercase

    "master":    "${rs_master}",
    "sales":     "${rs_sales}",
    "product":   "${rs_product}",
    "reporting": "${rs_reporting}",
}

Also Set the Output Schema

The output schema is the schema written into CREATE OR ALTER VIEW HERE.view_name. Change it on line 65:

copy
# The schema placeholder for the VIEW being created (output side)
output_schema_placeholder: str = "${os_reporting}"  # was ${os_bi_alefdw}

Result

After adding the mappings, this Redshift input:

copy
CREATE OR REPLACE VIEW reporting.vw_customer_orders AS
SELECT c.customer_id, o.order_id, p.product_name
FROM master.customer c
INNER JOIN sales.orders o ON c.customer_id = o.customer_id
INNER JOIN product.products p ON o.product_id = p.product_id;

Becomes this Fabric T-SQL output:

copy
CREATE OR ALTER VIEW ${os_reporting}.vw_customer_orders AS
SELECT c.customer_id, o.order_id, p.product_name
FROM ${rs_master}.customer c
INNER JOIN ${rs_sales}.orders o ON c.customer_id = o.customer_id
INNER JOIN ${rs_product}.products p ON o.product_id = p.product_id;
⚠ Rules to Remember
  • Keys in schema_placeholder_map must always be lowercase — "Master" won't match, "master" will.
  • Multiple source schemas can map to the same placeholder if they're the same logical database (e.g. dev and prod both → ${rs_bi_alefdw}).
  • Restart the server after editing settings.py — settings are loaded once at startup.

Placeholders Explained

Placeholders are the ${...} tokens that appear in generated Fabric T-SQL. They are resolved at deploy time by Flyway or your CI/CD pipeline.

The Three Placeholder Types

PlaceholderUsed InSettingMeaning
${schema} CREATE TABLE flyway_schema_placeholder The target schema for all tables — resolved by Flyway at migration time
${os_bi_alefdw} CREATE OR ALTER VIEW output_schema_placeholder The output schema where views live in Fabric — your warehouse schema
${rs_bi_alefdw} Inside VIEW bodies (FROM, JOIN) schema_placeholder_map Source read schema — where the underlying tables are in Fabric
${rs_alefdw} Inside VIEW bodies (JOIN on DW tables) schema_placeholder_map DW read schema — a second source database in Fabric

Naming Convention

PrefixMeaningExample
rs_Read Schema — source tables you query${rs_bi_alefdw}
os_Output Schema — where you write views/procs${os_bi_alefdw}
(none)Generic Flyway schema${schema}

Example Output

copy
-- TABLE DDL (uses ${schema})
IF OBJECT_ID('${schema}.student_login', 'U') IS NULL
BEGIN
    CREATE TABLE ${schema}.student_login (
        login_date_dw_id  BIGINT,
        school_dw_id      BIGINT
    );
END;

-- VIEW DDL (uses ${os_bi_alefdw} for the view, ${rs_bi_alefdw} for source)
CREATE OR ALTER VIEW ${os_bi_alefdw}.v_student_summary AS
SELECT * FROM ${rs_bi_alefdw}.student_login;

Function Mapping

All function transformations run in app/transformer/view_transformer.py. The pipeline is defined in transform_view() around line 55.

NVL(a, b)
→ ISNULL(a, b)
2-arg only. 3+ args → COALESCE
NVL(a, b, c)
→ COALESCE(a, b, c)
3+ args auto-detected
COALESCE(a, b)
→ ISNULL(a, b)
2-arg → ISNULL preferred in T-SQL
COALESCE(a, b, c)
→ COALESCE(a, b, c)
3+ args kept as COALESCE
DATE_TRUNC('week', x)
→ DATETRUNC(iso_week, x)
week → iso_week for ISO-8601
DATE_TRUNC('month', x)
→ DATETRUNC(month, x)
Arg order swapped
DATE_PART_YEAR(x)
→ DATEPART(YEAR, x)
Direct replacement
date_part(year, x)
→ DATEPART(YEAR, x)
Any date part supported
CURRENT_DATE
→ CONVERT(DATE, GETDATE())
Direct
CURRENT_TIMESTAMP
→ GETDATE()
Direct
TRUNC(SYSDATE) - 1
→ DATEADD(DAY,-1,CAST(GETDATE() AS DATE))
Compound pattern
expr::timestamp
→ CONVERT(DATETIME2(6), expr)
Backward-scanning :: parser
CAST(x AS type)
→ CONVERT(type, x)
Preferred T-SQL form
date(expr)
→ CONVERT(DATE, expr)
Postgres date() function
IS FALSE / IS TRUE
→ = 0 / = 1
Boolean expression rewrite
LISTAGG(col, ',')
→ STRING_AGG(col, ',')
DISTINCT not supported in STRING_AGG
INITCAP(expr)
→ UPPER(expr) + comment
No Title Case in Fabric

→ a + b
Pipe concat → + operator
CONVERT_TIMEZONE('UTC',tz,col)
→ AT TIME ZONE pattern

Materialized View → Stored Procedure

Fabric Warehouse does not support materialised views. The converter produces a stored procedure using the CTAS refresh pattern:

copy
-- Generated from: CREATE MATERIALIZED VIEW bi_alefdw.agg_foo_mv AS ...
CREATE OR ALTER PROCEDURE ${os_bi_alefdw}.usp_refresh_agg_foo AS
BEGIN
    DROP TABLE IF EXISTS ${os_bi_alefdw}.agg_foo_staging;
    CREATE TABLE ${os_bi_alefdw}.agg_foo_staging AS
    <original SELECT body>;
    DROP TABLE IF EXISTS ${os_bi_alefdw}.agg_foo;
    EXEC sp_rename 'agg_foo_staging', 'agg_foo';
END;
Extending the System

Add a New Schema Mapping

The most common customisation task. Takes under 60 seconds.

  1. Open app/core/settings.pyFind the schema_placeholder_map dict around line 57.
  2. Add your entryKey = source schema name (lowercase). Value = the ${placeholder} you want in the output.
  3. Set output_schema_placeholderAround line 65 — change this to the schema that will own the views in Fabric.
  4. Restart the serverCtrl+C to stop, then python run.py server again.
  5. Test itPaste a view that uses your new schema and verify the output uses your new placeholder.
copy
# app/core/settings.py

schema_placeholder_map: dict[str, str] = {
    "bi_alefdw":   "${rs_bi_alefdw}",
    "alefdw":      "${rs_alefdw}",

    # ← add your schemas below this line
    "your_schema": "${rs_your_schema}",
}

output_schema_placeholder: str = "${os_your_output_schema}"

Add a New Transformation Rule

To add a new Redshift → Fabric function or syntax conversion.

  1. Write the transformer functionAdd it to app/transformer/view_transformer.py. Every function has the same signature.
  2. Add it to the pipelineFind the pipeline list in transform_view() around line 55. Add a tuple: ("RULE_ID", _your_function).
  3. Add a unit testAdd a test in tests/unit/test_view_transformer.py.
  4. Run testspython run.py test --suite unit — all 74+ must pass.
copy
# app/transformer/view_transformer.py

# Step 1 — write the function
def _transform_my_function(sql: str) -> tuple[str, list[ConversionWarning]]:
    """MY_FUNC(x) → MY_TSQL_EQUIV(x)"""
    warnings: list[ConversionWarning] = []
    result = re.sub(r'\bMY_FUNC\s*\(', 'MY_TSQL_EQUIV(', sql, flags=re.IGNORECASE)
    return result, warnings


# Step 2 — add to pipeline list (in transform_view)
pipeline: list[tuple[str, Callable]] = [
    ...
    ("MY_FUNCTION_RULE", _transform_my_function),   # ← add here
    ...
]

Troubleshooting

Common errors and their fixes.

Setup Issues

⚠ ModuleNotFoundError: No module named 'pydantic'

Dependencies not installed. The venv is empty after unzipping.

python -m pip install -r requirements.txt
⚠ Activate.ps1 is not recognized

Either the venv doesn't exist yet, or PowerShell execution is blocked.

# Create venv first
python -m venv .venv

# Allow scripts for this session
Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned

# Then activate
.\.venv\Scripts\Activate.ps1
⚠ 'pip' is not recognized

On some Python 3.13/3.14 installs, pip isn't in PATH. Always use:

python -m pip install ...

Conversion Issues

⚠ Schema not being replaced with placeholder

The schema name isn't in schema_placeholder_map. Keys must be lowercase.

Edit app/core/settings.py and add the schema. Restart the server.

⚠ GROUP BY 1,2,3 not expanded

The SELECT column list couldn't be parsed (likely a complex CASE or nested subquery). A warning is added and the ordinal is kept. Manual expansion required.

❌ Parse error / FAILED status

The DDL statement couldn't be parsed at all. Common causes:

  • Missing semicolon between statements
  • Deeply nested or non-standard Redshift syntax
  • Statements with CREATE PROCEDURE / FUNCTION (not yet supported)

Check the warning message for the exact parse error. Split the file and test statements individually.
